# 架构与数据模型

## 1. 当前只保留的集合

个人 RAG 最小闭环只需要 6 类数据：

1. `users`
2. `knowledge_bases`
3. `documents`
4. `document_chunks`
5. `chat_sessions`
6. `chat_messages`

---

## 2. ID 统一规则

- DTO、JWT、路由参数中的 id 使用 `string`
- MongoDB schema 中的关联字段使用 `ObjectId`
- service 进入数据库查询前，统一把 `string` 转为 `ObjectId`
- 接口序列化输出继续返回 `string`

---

## 3. `knowledge_bases` 最小字段

只保留：

| 字段          | 类型     | 必填 | 说明       |
| ------------- | -------- | ---- | ---------- |
| `_id`         | ObjectId | 是   | 主键       |
| `userId`      | ObjectId | 是   | 所属用户   |
| `name`        | string   | 是   | 知识库名称 |
| `description` | string   | 否   | 知识库描述 |
| `createdAt`   | Date     | 是   | 创建时间   |
| `updatedAt`   | Date     | 是   | 更新时间   |

不再保留：

- `documentCount`
- `chunkCount`
- `sessionCount`
- `lastIndexedAt`

---

## 4. `documents` 最小字段

只保留：

| 字段              | 类型     | 必填 | 说明                 |
| ----------------- | -------- | ---- | -------------------- |
| `_id`             | ObjectId | 是   | 主键                 |
| `userId`          | ObjectId | 是   | 所属用户             |
| `knowledgeBaseId` | ObjectId | 是   | 所属知识库           |
| `sourceType`      | string   | 是   | `upload` 或 `editor` |
| `originalName`    | string   | 是   | 文档名称             |
| `storageKey`      | string   | 否   | 原文件对象存储路径   |
| `content`         | string   | 否   | 供切分使用的正文     |
| `extension`       | string   | 是   | 文件扩展名           |
| `mimeType`        | string   | 是   | 文件类型             |
| `size`            | number   | 是   | 文件大小或正文大小   |
| `createdAt`       | Date     | 是   | 创建时间             |
| `updatedAt`       | Date     | 是   | 更新时间             |

说明：

- 编辑器 Markdown 文档，`content` 必须直接保存原文
- `md` 上传文件，`content` 建议直接保存 UTF-8 文本
- `pdf / word` 上传文件，解析出正文后也建议写回 `content`

不再保留：

- `status`
- `chunkCount`
- `processedAt`
- `errorMessage`
- `splitterType`
- `splitConfig`
- 其他处理过程字段

---

## 5. `document_chunks` 最小字段

当前最小闭环只保留：

| 字段              | 类型     | 必填 | 说明       |
| ----------------- | -------- | ---- | ---------- |
| `_id`             | ObjectId | 是   | 主键       |
| `userId`          | ObjectId | 是   | 所属用户   |
| `knowledgeBaseId` | ObjectId | 是   | 所属知识库 |
| `documentId`      | ObjectId | 是   | 来源文档   |
| `sequence`        | number   | 是   | chunk 顺序 |
| `content`         | string   | 是   | chunk 文本 |
| `page`            | number   | 否   | PDF 等文档的页码定位 |
| `startIndex`      | number   | 否   | chunk 在原文中的起始位置 |
| `endIndex`        | number   | 否   | chunk 在原文中的结束位置 |
| `createdAt`       | Date     | 是   | 创建时间   |
| `updatedAt`       | Date     | 是   | 更新时间   |

说明：

- `page` 主要服务 PDF 来源引用
- `startIndex` 和 `endIndex` 主要服务原文定位与来源高亮
- 如果你使用 LangChain 并启用起始位置相关能力，可以直接映射到这两个字段

当前最小闭环先不放：

- `embedding`
- `headingPath`
- `metadata`

这些都不是当前“先跑通”必需项。

---

## 6. `POST /chat/ask` 最小输入输出

### 请求体

| 字段              | 类型   | 必填 | 说明             |
| ----------------- | ------ | ---- | ---------------- |
| `knowledgeBaseId` | string | 是   | 当前提问的知识库 |
| `sessionId`       | string | 否   | 继续已有会话时传入 |
| `question`        | string | 是   | 用户问题         |

### 响应体

| 字段        | 类型   | 必填 | 说明           |
| ----------- | ------ | ---- | -------------- |
| `sessionId` | string | 是   | 当前会话 ID    |
| `answer`    | string | 是   | AI 回答        |
| `sources`   | array  | 否   | 命中的来源片段 |

### `sources` 最小结构

| 字段            | 类型   | 必填 | 说明            |
| --------------- | ------ | ---- | --------------- |
| `documentId`    | string | 是   | 来源文档        |
| `documentName`  | string | 是   | 来源文档名      |
| `chunkSequence` | number | 是   | 来源 chunk 顺序 |
| `page`          | number | 否   | 来源页码        |
| `startIndex`    | number | 否   | 原文起始位置    |
| `endIndex`      | number | 否   | 原文结束位置    |

---

## 7. `chat_sessions` 最小字段

当前最小闭环只保留：

| 字段              | 类型     | 必填 | 说明       |
| ----------------- | -------- | ---- | ---------- |
| `_id`             | ObjectId | 是   | 主键       |
| `userId`          | ObjectId | 是   | 所属用户   |
| `knowledgeBaseId` | ObjectId | 是   | 所属知识库 |
| `title`           | string   | 是   | 会话标题   |
| `createdAt`       | Date     | 是   | 创建时间   |
| `updatedAt`       | Date     | 是   | 更新时间   |

说明：

- `chat_sessions` 不是 LangChain 原生对象，它只是业务层的会话容器
- `title` 是前端展示会话列表必需字段
- 当前最小闭环下，推荐用第一条 `human` 消息截断生成 `title`
- 后续如果要做 AI 自动总结标题，再单独扩展

---

## 8. `chat_messages` 最小字段

当前最小闭环只保留：

| 字段              | 类型     | 必填 | 说明 |
| ----------------- | -------- | ---- | --- |
| `_id`             | ObjectId | 是   | 主键 |
| `userId`          | ObjectId | 是   | 所属用户 |
| `knowledgeBaseId` | ObjectId | 是   | 所属知识库 |
| `sessionId`       | ObjectId | 是   | 所属会话 |
| `messageType`     | string   | 是   | LangChain 消息类型 |
| `content`         | string   | 是   | 消息文本 |
| `sequence`        | number   | 是   | 会话内顺序 |
| `sources`         | array    | 否   | 仅 AI 消息需要时保存来源快照 |
| `createdAt`       | Date     | 是   | 创建时间 |
| `updatedAt`       | Date     | 是   | 更新时间 |

### `messageType` 最小枚举

- `system`
- `human`
- `ai`
- `tool`

### 与 LangChain 的最小映射

- `system` -> `SystemMessage`
- `human` -> `HumanMessage`
- `ai` -> `AIMessage`
- `tool` -> `ToolMessage`

### 当前最小闭环实际接 LangChain 时只落这些参数

- `messageType`
- `content`

如果是 AI 回答，再额外保存：

- `sources`

当前最小闭环先不放：

- `id`
- `messageId`
- `responseMetadata`
- `usageMetadata`
- `toolCalls`
- `toolCallId`
- `artifact`

这些都不是“先跑通个人 RAG”必需项。

结论：

- LangChain 消息对象是运行时对象
- MongoDB 里只保存最小消息事实，不整包保存 LangChain 返回值

---

## 9. 当前最小数据流

```text
users
  -> knowledge_bases
      -> documents
          -> document_chunks
      -> chat_sessions
          -> chat_messages
              -> chat/ask
```
