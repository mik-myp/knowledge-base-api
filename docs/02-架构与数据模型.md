# 02 架构与数据模型

> 状态：规划中
>
> 适用范围：从产品交互、业务闭环和后端实现三个视角，重新梳理整个项目的数据模型。重点说明每张表为什么存在、最基础功能先需要哪些字段、后续还能补什么字段。

---

## 1. 先从最简单的业务闭环理解数据结构

如果只看“知识库问答”这四个字，很容易一上来就想做 RAG、向量检索、流式输出。但真正最小可用的产品闭环，其实是下面这条链路：

```text
注册 / 登录
  -> 创建知识库
  -> 上传文档并生成文档记录
  -> 展示当前用户下所有知识库的文档列表
  -> 把文档切成 chunk
  -> 在指定知识库里提问
  -> 保存会话和消息
```

这条链路决定了最小需要 6 张集合：

1. `users`
2. `knowledge_bases`
3. `documents`
4. `document_chunks`
5. `chat_sessions`
6. `chat_messages`

只有先把这 6 张集合理顺，后面的 RAG、Embedding、SSE 才有稳定基础。

---

## 2. 整体关系图

完整项目的核心归属关系如下：

```text
users
  └─ knowledge_bases
       ├─ documents
       │    └─ document_chunks
       └─ chat_sessions
            └─ chat_messages
```

对应业务关系：

1. 一个用户有多个知识库
2. 一个知识库有多个文档
3. 一个文档有多个 chunk
4. 一个知识库有多个问答会话
5. 一个问答会话有多条消息

换成更接近关系型数据库的表达：

| 主集合 | 从集合 | 关系 | 说明 |
| --- | --- | --- | --- |
| `users` | `knowledge_bases` | 1 对多 | 一个用户可以拥有多个知识库 |
| `knowledge_bases` | `documents` | 1 对多 | 一个知识库下可以有多份文档 |
| `documents` | `document_chunks` | 1 对多 | 一份文档会被拆成多个 chunk |
| `knowledge_bases` | `chat_sessions` | 1 对多 | 一个知识库可以有多个问答会话 |
| `chat_sessions` | `chat_messages` | 1 对多 | 一个会话由多条消息组成 |

## 3. 为什么要有这些表

对前端背景开发者来说，最难的不是记字段，而是理解“这张表到底帮谁解决问题”。

先按页面理解：

| 页面 / 交互 | 核心集合 | 为什么需要 |
| --- | --- | --- |
| 登录 / 注册 | `users` | 识别当前用户是谁 |
| 知识库列表 | `knowledge_bases` | 存知识库主记录与汇总字段 |
| 用户级文档列表 | `documents` | 统一展示当前用户所有知识库中的文档 |
| 文档检索基础 | `document_chunks` | 让文档内容可被检索 |
| 会话列表 | `chat_sessions` | 组织一轮轮问答 |
| 消息流 | `chat_messages` | 保存用户消息和助手消息 |

这里要注意：

- 一个页面可能依赖多张集合
- 一张集合也可能服务多个页面
- 表结构不是为页面硬拼出来的，而是为业务对象和查询场景服务的

---

## 4. Schema 设计总约定

为了让后续模块写法和当前仓库保持一致，推荐统一遵循这些约定：

- 全部使用 `@Schema({ collection, timestamps: true, versionKey: false })`
- 全部导出 `XxxDocument = HydratedDocument<Xxx>`
- 关联字段统一使用 `@Prop({ type: Types.ObjectId, ref: '...' })`
- 需要索引的地方，在 `SchemaFactory.createForClass(...)` 之后显式 `index(...)`
- `mongooseSerializePlugin` 通过 [app.module.ts](/Volumes/ZHITAI/project/knowledge-base-api/src/app.module.ts) 全局挂载，schema 文件里不重复 `plugin(...)`
- 拥有集合的模块自己负责 `MongooseModule.forFeature(...)`，不要在别的模块重复注册

这样做的好处：

- 写法统一
- `id` 序列化行为统一
- 新模块实现时可以直接照着已有风格扩展

## 5. 按从简单到复杂的顺序看每张表

### 5.1 `users`

当前状态：已实现

这张表解决的问题：

- 系统需要知道当前请求是谁
- 登录后需要保存 refreshToken 的哈希

它服务的基础功能：

- 注册
- 登录
- 刷新登录态
- 退出登录
- 获取当前用户

#### MVP 必需字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `_id` | ObjectId | 是 | 用户主键 |
| `email` | string | 是 | 登录邮箱，唯一 |
| `password` | string | 是 | 密码哈希 |
| `username` | string | 是 | 展示名称 |
| `status` | string | 是 | `active` / `disabled` |
| `refreshToken` | string | 否 | refreshToken 哈希 |

#### 后续可扩展字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `lastLoginAt` | Date | 否 | 最近登录时间 |
| `avatar` | string | 否 | 用户头像 |
| `bio` | string | 否 | 用户简介 |

#### 索引

- `email` 唯一索引

#### 设计说明

- `password` 存哈希，不存明文
- `refreshToken` 存哈希，便于服务端失效控制
- 第一阶段不需要引入复杂角色权限字段

### 5.2 `knowledge_bases`

当前状态：已实现

这张表解决的问题：

- 用户需要把文档和问答按主题分开管理
- 文档、chunk、会话都需要共同的归属容器

它服务的基础功能：

- 知识库列表
- 创建知识库
- 编辑知识库
- 删除知识库

#### MVP 必需字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `_id` | ObjectId | 是 | 知识库主键 |
| `userId` | ObjectId | 是 | 所属用户 |
| `name` | string | 是 | 知识库名称 |
| `description` | string | 否 | 知识库说明 |
| `documentCount` | number | 是 | 文档数量缓存 |
| `chunkCount` | number | 是 | chunk 数量缓存 |
| `sessionCount` | number | 是 | 会话数量缓存 |

#### 后续可扩展字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `lastIndexedAt` | Date | 否 | 最近一次索引完成时间 |
| `cover` | string | 否 | 知识库封面 |
| `visibility` | string | 否 | 后续做共享时再补 |

#### 索引

- `userId`
- `{ userId, updatedAt: -1 }`

#### 设计说明

- `documentCount`、`chunkCount`、`sessionCount` 都是缓存字段，不是唯一数据来源
- 第一阶段先把这些缓存字段建好，后续新增文档、切片、会话时再由 service 维护
- 这样知识库列表页不需要每次临时聚合统计

### 5.3 `documents`

当前状态：规划中

这张表解决的问题：

- 前端需要展示“当前用户下所有知识库中的全部文档”
- 后端需要知道一份文档属于哪个知识库、是什么类型、现在处理到哪一步

它服务的基础功能：

- 用户级文档列表
- 知识库内文档列表
- 文档上传后的元数据展示
- 文档状态展示

#### MVP 必需字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `_id` | ObjectId | 是 | 文档主键 |
| `userId` | ObjectId | 是 | 所属用户 |
| `knowledgeBaseId` | ObjectId | 是 | 所属知识库 |
| `originalName` | string | 是 | 原始文件名 |
| `extension` | string | 是 | 原始扩展名，例如 `pdf`、`docx` |
| `fileType` | string | 是 | 展示级文件类型，例如 `pdf`、`word`、`markdown` |
| `mimeType` | string | 是 | MIME 类型 |
| `size` | number | 是 | 文件大小，单位字节 |
| `status` | string | 是 | `pending/processing/ready/failed` |

#### 后续可扩展字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `statusMessage` | string | 否 | 当前状态说明 |
| `chunkCount` | number | 否 | 当前文档 chunk 数量缓存 |
| `processedAt` | Date | 否 | 文档处理完成时间 |
| `storageKey` | string | 否 | 对象存储路径 |
| `errorMessage` | string | 否 | 失败原因 |

#### 推荐 `fileType`

- `pdf`
- `word`
- `markdown`
- `text`
- `spreadsheet`
- `presentation`
- `other`

#### 推荐 `status`

- `pending`
- `processing`
- `ready`
- `failed`

#### 索引

- `userId`
- `knowledgeBaseId`
- `{ userId, updatedAt: -1 }`
- `{ knowledgeBaseId, updatedAt: -1 }`

#### 设计说明

- `extension` 保留原始文件扩展名，用于技术层处理
- `fileType` 用于前端展示和筛选
- 第一阶段不要求先接对象存储，只要能把文档主记录跑通即可

### 5.4 `document_chunks`

当前状态：规划中

这张表解决的问题：

- 模型真正检索的是 chunk，不是整份文档
- 文档内容需要拆成适合检索和引用展示的最小单元

它服务的基础功能：

- 检索上下文
- 引用来源
- 文档索引重建

#### MVP 必需字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `_id` | ObjectId | 是 | chunk 主键 |
| `userId` | ObjectId | 是 | 所属用户 |
| `knowledgeBaseId` | ObjectId | 是 | 所属知识库 |
| `documentId` | ObjectId | 是 | 来源文档 |
| `sequence` | number | 是 | 文档内顺序 |
| `content` | string | 是 | chunk 文本 |
| `charCount` | number | 是 | 字符数 |

#### 后续可扩展字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `tokenCount` | number | 否 | token 数 |
| `embedding` | number[] | 否 | 向量字段 |
| `metadata` | object | 否 | 额外解析信息 |

#### 索引

- `documentId`
- `knowledgeBaseId`
- `{ documentId, sequence }` 唯一索引

#### 设计说明

- `sequence` 用于保证文档切片的顺序稳定
- 第一阶段不需要强制把 `embedding` 放进去，可以先把 chunk 存下来
- 后续真正接向量检索时再补向量字段和索引

### 5.5 `chat_sessions`

当前状态：规划中

这张表解决的问题：

- 聊天页需要有“会话”这个概念
- 同一个知识库里多轮问答要能区分和回放

它服务的基础功能：

- 会话列表
- 进入某个历史会话

#### MVP 必需字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `_id` | ObjectId | 是 | 会话主键 |
| `userId` | ObjectId | 是 | 所属用户 |
| `knowledgeBaseId` | ObjectId | 是 | 所属知识库 |
| `title` | string | 是 | 会话标题 |
| `messageCount` | number | 是 | 消息总数 |
| `lastMessageAt` | Date | 否 | 最近消息时间 |

#### 后续可扩展字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `summary` | string | 否 | 会话摘要 |
| `pinned` | boolean | 否 | 是否置顶 |

#### 索引

- `userId`
- `knowledgeBaseId`
- `{ knowledgeBaseId, updatedAt: -1 }`

#### 设计说明

- `title` 第一版可以直接取第一条用户问题的截断文本
- `messageCount` 也是缓存字段，便于会话列表展示

### 5.6 `chat_messages`

当前状态：规划中

这张表解决的问题：

- 一轮问答不是一个结果，而是一连串消息
- 历史消息需要可回放、可排序、可展示引用来源

它服务的基础功能：

- 消息流展示
- 上下文持久化
- 历史消息回放

#### MVP 必需字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `_id` | ObjectId | 是 | 消息主键 |
| `userId` | ObjectId | 是 | 所属用户 |
| `knowledgeBaseId` | ObjectId | 是 | 所属知识库 |
| `sessionId` | ObjectId | 是 | 所属会话 |
| `role` | string | 是 | `user/assistant/system` |
| `content` | string | 是 | 消息内容 |
| `sequence` | number | 是 | 会话内顺序 |

#### 后续可扩展字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `sources` | object[] | 否 | 引用来源快照 |
| `model` | string | 否 | 回答所用模型 |
| `usage` | object | 否 | token 统计 |
| `finishReason` | string | 否 | 流式结束原因 |

#### 索引

- `sessionId`
- `{ sessionId, sequence }` 唯一索引
- `{ userId, knowledgeBaseId }`

#### 设计说明

- `role` 是区分用户、助手、系统消息的核心字段
- `sequence` 用来恢复消息顺序
- `sources` 适合做快照，而不是完全依赖后续回查

## 6. 为什么要冗余这些字段

### 6.1 为什么很多表都要有 `userId`

在 `documents`、`document_chunks`、`chat_sessions`、`chat_messages` 中冗余 `userId`，好处是：

- 权限过滤简单
- 删除用户数据时更容易级联
- 不需要每次回查父集合再确认归属

### 6.2 为什么 `document_chunks` 和 `chat_messages` 还要有 `knowledgeBaseId`

继续冗余 `knowledgeBaseId`，好处是：

- 按知识库维度查询更直接
- 检索时不需要每次从 `documentId` 或 `sessionId` 回表
- 删除知识库时更容易做级联删除

### 6.3 为什么知识库要有汇总字段

`documentCount`、`chunkCount`、`sessionCount` 的作用是：

- 列表页快速展示
- 减少频繁聚合统计
- 让前端直接拿到概览信息

这类字段的本质是缓存，不是唯一真相，所以必须由业务层维护。

## 7. 文档列表接口为什么必须独立设计

前端有一个明确页面，需要：

> 以列表形式展示当前用户下所有知识库的所有文档，并按文件类型区分。

因此更适合提供独立接口：

- `GET /documents`

而不是把文档列表强行塞进知识库详情。

第一版推荐支持：

- 分页
- 按 `knowledgeBaseId` 筛选
- 按 `fileType` 筛选
- 按 `status` 筛选

第一版首屏字段建议返回：

- `id`
- `knowledgeBaseId`
- `knowledgeBaseName`
- `originalName`
- `extension`
- `fileType`
- `mimeType`
- `size`
- `status`
- `updatedAt`

## 8. 删除策略

本项目当前推荐采用“硬删除级联”。

### 8.1 删除知识库时

应删除：

- 当前 `knowledge_bases` 记录
- 该知识库下所有 `documents`
- 该知识库下所有 `document_chunks`
- 该知识库下所有 `chat_sessions`
- 该知识库下所有 `chat_messages`

### 8.2 删除文档时

应删除：

- 当前 `documents` 记录
- 对应所有 `document_chunks`

### 8.3 删除文档是否保留聊天历史

第一阶段可以先按“知识库级硬删除”为主，不提前把历史保留策略做复杂。

如果后续产品明确要求“即使删文档也保留历史答案与引用快照”，再在 `chat_messages.sources` 设计基础上补更精细规则。

## 9. MongooseModule 的使用原则

后续模块建议固定遵循这条规则：

- `UsersModule` 注册 `users`
- `KnowledgeBasesModule` 注册 `knowledge_bases`
- `DocumentsModule` 注册 `documents` 和 `document_chunks`
- `ChatModule` 注册 `chat_sessions` 和 `chat_messages`

也就是说：

- 谁拥有集合，谁在自己的模块里 `MongooseModule.forFeature(...)`
- 其他模块通过导入模块协作，不直接重复注册别人拥有的 schema
- `rag`、`storage` 这类能力型模块，在这一轮不单独拥有 MongoDB 集合
