# 架构与数据模型

## 1. 这次重写的依据

这份文档不是按“理想中的项目”写，而是按当前两个仓库的真实状态重写：

- 前端 `/Volumes/ZHITAI/project/knowledge-front-end`
  - 已安装 `@ant-design/x`、`@ant-design/x-sdk`
  - 已有 AI 页骨架：`AILayout`、`ChatSide`、`ChatList`、`ChatSender`
  - 当前仍是占位接法，`XRequest('https://api.example.com/chat')` 还没接真实后端
- 后端 `/Volumes/ZHITAI/project/knowledge-base-api`
  - 已实现 `users`、`knowledge_bases`、`documents`、`storage`
  - 已实现 `document_chunks`
  - 已实现 `document_chunk_vectors` 的写入、删除、向量检索
  - 上传链路已经是“任一环节失败则整体回滚”

所以这份文档的目标不是继续讨论“要不要这么做”，而是把前端页面需要的数据、后端 LangChain/RAG 需要的数据、数据库应该存什么，统一梳理清楚。

---

## 2. 职责边界

### 前端 `Ant Design X + X SDK`

前端负责：

- 展示会话列表
- 展示消息列表
- 管理输入框草稿、发送中状态、会话切换
- 调用后端接口
- 展示来源引用 `sources`

前端不负责：

- 直接调模型
- 直接访问向量库
- 拼 RAG prompt
- 保存最终会话事实

### 后端 `Nest + LangChain`

后端负责：

- 鉴权
- 知识库归属校验
- 文档上传、切片、向量化
- 会话创建与消息持久化
- 向量检索与提示词组装
- 调用 LangChain / 模型
- 返回前端可直接消费的 DTO

### 一个原则

`Ant Design X SDK` 是前端消息流管理层，不是模型调用层。  
前端可以用 `useXChat`、`useXConversations`、`XRequest`，但请求目标必须是自己的 Nest 接口，而不是模型厂商接口。

---

## 3. 需要保留的集合

当前这套 `AntDX + LangChain` 架构，最少需要 7 张业务集合和 1 张原始向量集合：

1. `users`
2. `knowledge_bases`
3. `documents`
4. `document_chunks`
5. `document_chunk_vectors`
6. `chat_sessions`
7. `chat_messages`

其中：

- `document_chunks` 是业务表，负责原文切片事实
- `document_chunk_vectors` 是向量检索表，负责 `$vectorSearch`
- `chat_sessions` 是前端会话列表容器
- `chat_messages` 是 LangChain 消息持久化表

---

## 4. ID 与类型规则

- 接口层统一返回 `string` 类型 id
- Mongoose schema 里的关联字段使用 `ObjectId`
- `document_chunk_vectors` 当前直接走 Mongo 原生 collection，里面的 `userId`、`knowledgeBaseId`、`documentId` 继续保存为字符串
- 进入数据库查询前，service 层负责统一转换

这样做的原因很直接：

- 业务表用 `ObjectId`，便于和 Mongoose 协作
- 向量表用字符串过滤，便于 `$vectorSearch.filter`

## 5. 表结构总览

### 5.1 `knowledge_bases`

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `_id` | ObjectId | 是 | 主键 |
| `userId` | ObjectId | 是 | 所属用户 |
| `name` | string | 是 | 知识库名称 |
| `description` | string | 否 | 描述 |
| `createdAt` | Date | 是 | 创建时间 |
| `updatedAt` | Date | 是 | 更新时间 |

这张表只表达“知识库容器”。

不要往里塞：

- `documentCount`
- `chunkCount`
- `sessionCount`
- `lastIndexedAt`

这些都可以后算，先不要污染主表。

### 5.2 `documents`

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `_id` | ObjectId | 是 | 主键 |
| `userId` | ObjectId | 是 | 所属用户 |
| `knowledgeBaseId` | ObjectId | 是 | 所属知识库 |
| `sourceType` | `upload \| editor` | 是 | 来源类型 |
| `originalName` | string | 是 | 原始文件名或编辑器文档名 |
| `storageKey` | string | 否 | R2/对象存储路径 |
| `content` | string | 否 | 原始正文 |
| `extension` | string | 是 | 扩展名 |
| `mimeType` | string | 是 | MIME 类型 |
| `size` | number | 是 | 文件大小或正文字节数 |
| `createdAt` | Date | 是 | 创建时间 |
| `updatedAt` | Date | 是 | 更新时间 |

设计原则：

- `documents` 是文档主记录，不负责存切片
- 只要能拿到正文，优先把正文写回 `content`
- `editor` 创建的 Markdown 文档，`content` 必须保存完整原文
- `md/txt` 上传，`content` 直接保存 UTF-8 文本
- `pdf/docx` 由后端解析后参与切片，但当前实现没有把解析后的正文反写回 `documents.content`

### 5.3 `document_chunks`

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `_id` | ObjectId | 是 | 主键 |
| `userId` | ObjectId | 是 | 所属用户 |
| `knowledgeBaseId` | ObjectId | 是 | 所属知识库 |
| `documentId` | ObjectId | 是 | 来源文档 |
| `sequence` | number | 是 | 文档内顺序 |
| `content` | string | 是 | chunk 文本 |
| `page` | number | 否 | PDF 页码 |
| `startIndex` | number | 否 | 在原文中的起始偏移 |
| `endIndex` | number | 否 | 在原文中的结束偏移 |
| `createdAt` | Date | 是 | 创建时间 |
| `updatedAt` | Date | 是 | 更新时间 |

这张表是“引用事实表”，给两个场景用：

- 前端展示来源位置
- 后端做删除、重建、审计

当前实现规则：

- `md/txt/docx` 手动计算 `startIndex` / `endIndex`
- `pdf` 保留 `page`
- 当前实现里，`pdf` 也会在页内继续计算 `startIndex` / `endIndex`

### 5.4 `document_chunk_vectors`

这是原始 Mongo collection，不走 Mongoose schema。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `_id` | ObjectId | 是 | 主键 |
| `userId` | string | 是 | 用户 id，字符串 |
| `knowledgeBaseId` | string | 是 | 知识库 id，字符串 |
| `documentId` | string | 是 | 文档 id，字符串 |
| `documentName` | string | 是 | 文档名快照 |
| `sequence` | number | 是 | chunk 顺序 |
| `text` | string | 是 | 向量化文本 |
| `embedding` | number[] | 是 | 向量 |
| `page` | number | 否 | PDF 页码 |
| `startIndex` | number | 否 | 起始偏移 |
| `endIndex` | number | 否 | 结束偏移 |

这张表只服务于检索，不承担业务聚合职责。

必须注意：

- 向量表里的文本字段叫 `text`，不是 `content`
- 当前 `OPENAI_EMBEDDING_BATCH_SIZE` 最大只能到 `10`
- `$vectorSearch.filter` 用的是字符串字段过滤，不要改成 `ObjectId`

### 5.5 `chat_sessions`

这张表现在后端还没落代码，但从前端 `Conversations` 的需求看，它必须存在。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `_id` | ObjectId | 是 | 主键 |
| `userId` | ObjectId | 是 | 所属用户 |
| `knowledgeBaseId` | ObjectId | 是 | 当前会话绑定的知识库 |
| `title` | string | 是 | 会话标题 |
| `createdAt` | Date | 是 | 创建时间 |
| `updatedAt` | Date | 是 | 最近活跃时间 |

这里故意不加：

- `group`
- `loading`
- `active`
- `messageCount`

原因：

- `group` 是前端展示字段，可由 `updatedAt` 推导为“今天/昨天/更早”
- `loading`、`active` 是页面状态，不应落库
- `messageCount` 不是当前最小闭环必需字段

### 5.6 `chat_messages`

这张表也还没落代码，但已经可以从前端消息列表和 LangChain 消息模型反推出最小结构。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `_id` | ObjectId | 是 | 主键 |
| `userId` | ObjectId | 是 | 所属用户 |
| `knowledgeBaseId` | ObjectId | 是 | 所属知识库 |
| `sessionId` | ObjectId | 是 | 所属会话 |
| `messageType` | `system \| human \| ai \| tool` | 是 | LangChain 消息类型 |
| `content` | string | 是 | 消息文本 |
| `sequence` | number | 是 | 会话内顺序 |
| `sources` | array | 否 | AI 回复来源快照 |
| `createdAt` | Date | 是 | 创建时间 |
| `updatedAt` | Date | 是 | 更新时间 |

`sources` 结构建议固定为：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `documentId` | ObjectId | 是 | 来源文档 |
| `documentName` | string | 是 | 文档名快照 |
| `chunkSequence` | number | 是 | 来源 chunk 顺序 |
| `page` | number | 否 | PDF 页码 |
| `startIndex` | number | 否 | 起始偏移 |
| `endIndex` | number | 否 | 结束偏移 |
| `score` | number | 否 | 命中分数快照 |

## 6. 前端组件需要什么字段

### 6.1 `Conversations` 组件

前端当前组件只真正需要：

- `key`
- `label`
- `group`

推荐后端实际返回：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | string | sessionId |
| `title` | string | 会话标题 |
| `knowledgeBaseId` | string | 所属知识库 |
| `updatedAt` | string | 最近活跃时间 |

然后前端映射：

- `key = id`
- `label = title`
- `group =` 基于 `updatedAt` 计算“今天/昨天/更早”

不要把 `group` 存数据库。

### 6.2 `Bubble.List` 组件

前端消息项真正需要：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | string | messageId |
| `role` | `user \| assistant` | 展示角色 |
| `content` | string | 消息文本 |
| `status` | `success \| loading \| error` | 前端状态 |
| `sources` | array | AI 回复来源 |

数据库到前端的映射建议固定：

- `human -> user`
- `ai -> assistant`
- `system -> 默认不展示`
- `tool -> 默认不展示，后续需要再扩`

### 6.3 `Sender + Suggestion`

前端当前 `ChatSender` 里最值得保留的是“知识库选择”的交互位。

因此页面级状态至少要有：

- `activeKnowledgeBaseId`
- `activeSessionId`
- `draftMessage`

不要让会话跨知识库复用。  
一条 `chat_session` 必须只绑定一个 `knowledgeBaseId`。

## 7. 推荐接口契约

### 7.1 会话列表

`GET /chat/sessions?knowledgeBaseId=xxx`

响应：

```json
{
  "dataList": [
    {
      "id": "session_1",
      "title": "JWT 鉴权怎么接到 AntDX 页面里",
      "knowledgeBaseId": "kb_1",
      "updatedAt": "2026-03-23T10:00:00.000Z"
    }
  ],
  "total": 1
}
```

### 7.2 消息列表

`GET /chat/messages?sessionId=xxx`

响应：

```json
{
  "dataList": [
    {
      "id": "msg_1",
      "messageType": "human",
      "content": "JWT 鉴权怎么接到 AntDX 页面里？",
      "sequence": 0
    },
    {
      "id": "msg_2",
      "messageType": "ai",
      "content": "建议由前端只调 Nest 接口，后端统一走 LangChain。",
      "sequence": 1,
      "sources": [
        {
          "documentId": "doc_1",
          "documentName": "架构说明.md",
          "chunkSequence": 3,
          "startIndex": 120,
          "endIndex": 260,
          "score": 0.93
        }
      ]
    }
  ]
}
```

### 7.3 单次问答

`POST /chat/ask`

请求体：

```json
{
  "knowledgeBaseId": "kb_1",
  "sessionId": "session_1",
  "question": "JWT 鉴权怎么接到 AntDX 页面里？",
  "topK": 5
}
```

响应体：

```json
{
  "sessionId": "session_1",
  "answer": "建议由前端只调 Nest 接口，后端统一走 LangChain。",
  "sources": [
    {
      "documentId": "doc_1",
      "documentName": "架构说明.md",
      "chunkSequence": 3,
      "startIndex": 120,
      "endIndex": 260,
      "score": 0.93
    }
  ]
}
```

## 8. 这套模型为什么能同时服务 AntDX 和 LangChain

因为它刻意把三层需求拆开了：

- `documents / document_chunks / document_chunk_vectors` 服务 RAG 检索
- `chat_sessions / chat_messages` 服务前端会话与消息展示
- DTO 映射层负责把 `messageType` 转成 `Bubble.List` 需要的 `role`

不要做下面这两件事：

- 不要让前端消息结构直接等于 LangChain 返回对象
- 不要让数据库结构直接等于 AntDX 组件 props

数据库存事实，前端吃 DTO，LangChain 只参与执行链路。这样边界最稳。
