## 1. 关系总览

本项目完整目标中的核心关系如下：

```text
user
  └─ knowledge_bases
       ├─ documents
       │    └─ document_chunks
       └─ chat_sessions
            └─ chat_messages
```

对应的业务关系是：

1. 一个用户有多个知识库
2. 一个知识库有多个文档
3. 一个文档有多个 chunk
4. 一个知识库有多个问答会话
5. 一个问答会话有多个对话消息

## 4. 冗余字段原则

这里会刻意冗余一部分归属字段，而不是所有查询都依赖跨集合查找。

### 4.1 为什么要冗余 `userId`

在 `documents`、`document_chunks`、`chat_sessions`、`chat_messages` 中冗余 `userId`，好处是：

- 权限过滤简单
- 按用户查询效率更高
- 级联删除更直观

### 4.2 为什么还要冗余 `knowledgeBaseId`

在 `document_chunks` 和 `chat_messages` 中继续冗余 `knowledgeBaseId`，好处是：

- 面向知识库维度筛选更直接
- 后续检索时不需要每次回表找归属

## 5. 文档列表接口为什么独立设计

前端有一个页面，需要“以列表形式展示当前用户下所有知识库下的所有文档，并按文件类型区分”。

因此后端接口应当独立提供用户级文档列表，而不是强耦合在知识库详情中拼装。

推荐接口方向：

- `GET /documents`

第一版能力：

- 分页
- 按 `knowledgeBaseId` 筛选
- 按 `fileType` 筛选

第一版首屏字段：

- `id`
- `knowledgeBaseId`
- `knowledgeBaseName`
- `originalName`
- `extension`
- `fileType`
- `mimeType`
- `size`
- `status`
- `updatedAt`

## 6. 删除策略

本项目当前定稿为“硬删除级联”。

### 6.1 删除知识库时

应删除：

- `knowledge_bases`
- 该知识库下所有 `documents`
- 该知识库下所有 `document_chunks`
- 该知识库下所有 `chat_sessions`
- 该知识库下所有 `chat_messages`

### 6.2 删除文档时

应删除：

- 当前 `documents` 记录
- 对应所有 `document_chunks`
