# 00 项目产品与业务总览

> 状态：产品文档
>
> 适用范围：帮助从中途加入的开发者，尤其是纯前端背景且首次接触后端的同学，快速理解项目要做什么、用户如何使用、交互与业务如何流转。

---

## 1. 这个项目在做什么

一句话说明：

> 让用户把自己的资料整理成知识库，然后像聊天一样提问，系统基于用户自己的资料检索后给出回答。

这个项目不是单纯“接一个大模型接口”，而是要完成一条完整产品链路：

1. 用户登录系统
2. 用户创建知识库
3. 用户向知识库里加入文档
4. 系统把文档处理成可检索的数据
5. 用户在指定知识库里提问
6. 系统基于该知识库内容回答
7. 问答过程保存为会话和消息

---

## 2. 目标用户与核心价值

### 2.1 目标用户

- 个人学习者：整理课程资料、笔记、项目文档
- 内容生产者：整理写作素材、采访资料、行业文档
- 面试准备者：整理题库、项目经验、复盘材料

### 2.2 用户价值

- 不需要手动翻大量资料
- 可以在指定知识库内定向提问
- 能把知识组织成长期可复用资产
- 后续回答具备来源可追溯能力

---

## 3. 用户角色

当前产品只考虑一个核心角色：

- 普通登录用户

暂不考虑：

- 管理员后台
- 团队协作权限
- 多租户企业空间

这样做是为了让主线保持清晰，先把“单用户知识库问答”跑通。

---

## 4. 核心页面与交互逻辑

### 4.1 登录 / 注册页

用户动作：

- 注册账号
- 登录
- 刷新登录态
- 退出登录

后端职责：

- 创建用户
- 校验邮箱和密码
- 签发 accessToken / refreshToken
- 返回当前用户信息

### 4.2 知识库列表页

用户动作：

- 查看自己所有知识库
- 创建新知识库
- 编辑知识库名称和描述
- 删除知识库

页面核心信息：

- 知识库名称
- 描述
- 文档数
- chunk 数
- 会话数
- 最近索引时间

后端职责：

- 只返回当前用户自己的知识库
- 按更新时间排序
- 维护知识库级汇总字段

### 4.3 用户级文档列表页

这是当前产品里非常关键的一页。

用户动作：

- 查看当前用户下所有知识库中的全部文档
- 按知识库筛选
- 按文件类型筛选
- 查看文档状态

页面核心信息：

- 文档名称
- 所属知识库
- 文件类型
- 文件大小
- 当前状态
- 最近更新时间

这页存在的意义：

- 用户不需要先点进每个知识库逐个看文档
- 适合做统一管理和排错

### 4.4 知识库问答页

用户动作：

- 选择一个知识库
- 输入问题
- 查看回答
- 查看回答引用来源
- 查看历史会话

页面核心信息：

- 当前知识库
- 当前会话
- 用户消息
- 助手消息
- 引用来源

### 4.5 会话历史页

用户动作：

- 查看某个知识库下的会话列表
- 点击进入某个会话
- 查看消息流

后端职责：

- 会话和消息都必须严格归属于当前用户和当前知识库

## 5. 业务闭环

从产品视角看，完整业务闭环是：

```text
注册/登录
  -> 创建知识库
  -> 导入文档
  -> 文档处理为 chunk
  -> 在知识库中提问
  -> 返回回答与引用
  -> 保存会话与消息
  -> 用户持续复用知识库
```

这个闭环里有三个关键对象：

- 知识库：内容容器
- 文档：原始资料
- 会话：使用过程

### 5.1 最小可用闭环

如果要先做出第一版最基础可用功能，建议只盯住这条最短闭环：

```text
注册 / 登录
  -> 创建知识库
  -> 创建文档主记录
  -> 展示用户级文档列表
  -> 文档切片
  -> 创建会话
  -> 保存消息
```

这条链路的关键目标不是一次把所有 AI 能力做满，而是先保证：

- 用户能拥有自己的数据
- 数据归属关系稳定
- 前端页面有稳定接口可用
- 后续 RAG 可以建立在已有数据之上

### 5.2 页面动作到数据写入的最小映射

| 页面动作 | 后端动作 | 主要写入集合 |
| --- | --- | --- |
| 用户注册 | 创建账号 | `users` |
| 创建知识库 | 创建知识库主记录 | `knowledge_bases` |
| 上传文档 | 创建文档主记录 | `documents` |
| 文档完成切片 | 写入 chunk | `document_chunks` |
| 用户首次提问 | 创建会话 + 保存消息 | `chat_sessions`、`chat_messages` |
| 助手返回回答 | 保存助手消息 | `chat_messages` |

## 6. 核心业务逻辑

### 6.1 知识库业务逻辑

- 一个用户可以有多个知识库
- 一个知识库只属于一个用户
- 知识库是文档和问答的共同归属容器

### 6.2 文档业务逻辑

- 一个知识库下可以有多份文档
- 文档需要支持文件类型归类
- 文档需要有状态，表示当前是否已经可用

### 6.3 检索业务逻辑

- 用户提问时，必须限定在当前知识库范围内检索
- 不能把不同知识库甚至不同用户的数据混在一起检索

### 6.4 会话业务逻辑

- 一个知识库可以有多个问答会话
- 一个会话由多条消息组成
- 消息需要保留顺序和角色

---

## 7. 关键状态流转

### 7.1 知识库创建流转

```text
用户填写名称/描述
  -> 后端校验
  -> 创建知识库主记录
  -> 初始化 documentCount/chunkCount/sessionCount
  -> 返回知识库详情
```

### 7.2 文档处理流转

当前推荐第一版状态：

```text
pending -> processing -> ready
pending -> processing -> failed
```

它表示：

- `pending`：文档记录已创建，待处理
- `processing`：文档正在解析、切片或索引
- `ready`：文档已经可用于检索
- `failed`：处理失败，需要查看原因

### 7.3 问答流转

```text
用户选择知识库并发问
  -> 系统检索相关 chunk
  -> 模型生成回答
  -> 返回答案与引用
  -> 写入会话和消息
```

## 8. 页面、接口、数据的映射关系

| 页面 / 交互 | 后端接口 | 核心集合 |
| --- | --- | --- |
| 注册 | `POST /users/register` | `users` |
| 登录 | `POST /users/login` | `users` |
| 当前用户 | `GET /users/me` | `users` |
| 知识库列表 | `GET /knowledge-bases` | `knowledge_bases` |
| 创建知识库 | `POST /knowledge-bases` | `knowledge_bases` |
| 更新知识库 | `PATCH /knowledge-bases/:id` | `knowledge_bases` |
| 删除知识库 | `DELETE /knowledge-bases/:id` | `knowledge_bases` |
| 用户级文档列表 | `GET /documents` | `documents` |
| 上传文档 | `POST /documents/upload` | `documents`、`document_chunks` |
| 发起问答 | `POST /chat/ask` | `chat_sessions`、`chat_messages` |
| 会话消息流 | `GET /chat/sessions/:id/messages` | `chat_messages` |

这里要特别注意：

- 上表里 `users` 和 `knowledge_bases` 是当前已实现
- `documents` 和 `chat` 相关接口是后续规划

---

## 9. 为什么这个项目适合前端开发者理解后端

因为这条主线比较完整，而且每一步都有明确的输入和输出：

- 登录：前端最熟悉
- 列表与详情：标准接口组织
- 文档处理：开始接触状态流转和异步业务
- 问答：开始接触检索、上下文和消息持久化

换句话说，这个项目能让前端背景开发者逐步理解：

- 请求是怎么进入后端的
- 后端为什么要拆 DTO、Service、Schema
- 为什么一个页面背后会对应多张集合和多层逻辑
