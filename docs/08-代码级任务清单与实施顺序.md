# 08 代码级任务清单与实施顺序

> 状态：实施清单
>
> 文档定位：把当前这套文档方案进一步落成“可执行的代码任务列表”，严格按 `documents -> document_chunks -> chat_sessions/chat_messages -> chat/ask -> chat/stream` 的顺序推进。

## 1. 这份文档解决什么问题

前面的文档已经把这几件事讲清楚了：

- 最小闭环应该怎么设计
- `documents / document_chunks / chat_sessions / chat_messages` 应该有哪些字段
- LangChain 在文档切分和对话里应该扮演什么角色

但如果准备真正开始写代码，还需要一份更落地的任务清单，明确：

- 先改哪些文件
- 每一阶段要交付哪些接口和 schema
- 现有代码里哪些地方要重构，哪些地方可以复用
- 每一阶段完成后怎么验收

这份文档就是为这个目的准备的。

---

## 2. 当前仓库的现实情况

在开始进入任务拆分前，先明确当前代码现实：

- `users`、`knowledge_bases` 已经存在
- `storage` 已经存在
- 仓库里已经有一个 `src/documnets/` 模块，但命名存在明显拼写问题
- 当前 `documnets` 模块和文档中的最终设计并不完全一致

因此后续实现不要机械地“在旧代码上继续堆字段”，而应该先做一次收口。

### 2.1 推荐先做的命名决策

当前仓库实际路径是：

```text
src/documnets/
  documnets.controller.ts
  documnets.module.ts
  documnets.service.ts
  schemas/documnet.schema.ts
```

推荐做法：

1. 先把 `documnets` 重命名为 `documents`
2. 再开始补 `document_chunks`、`chat`、`RAG`

原因：

- 后续所有文档、类型、DTO、测试都会围绕 `documents` 命名
- 如果继续带着这个拼写问题推进，后面修改成本只会更高

如果当前你不想先做重命名，也可以先在旧路径上继续实现，但这只是过渡方案，不是最终推荐方案。

---

## 3. 总体实施顺序

严格建议按这个顺序推进：

1. `documents`
2. `document_chunks`
3. `chat_sessions / chat_messages`
4. `chat/ask`
5. `chat/stream`

原因：

- `documents` 解决最直接的内容管理需求
- `document_chunks` 让文档真正可检索
- `chat_sessions / chat_messages` 让问答过程可持久化
- `chat/ask` 先用 JSON 问答跑通检索与落库
- `chat/stream` 最后再补流式复杂度

---

## 4. 阶段一：`documents`

### 4.1 目标

完成文档主记录层，支撑：

- 用户级文档列表
- 文件上传文档主记录
- Markdown 编辑器纯文本主记录
- 文档状态流转

### 4.2 本阶段交付

- `documents` schema 对齐最新文档设计
- `DocumentsModule`
- `GET /documents`
- `GET /documents/:id`
- `POST /documents/upload`
- `POST /documents/editor`
- `DELETE /documents/:id`

### 4.3 需要改动的代码

推荐优先处理这些文件：

```text
src/app.module.ts
src/documnets/
src/storage/storage.service.ts
```

如果先做重命名，目标路径应调整为：

```text
src/documents/
  dto/
  schemas/
  documents.controller.ts
  documents.module.ts
  documents.service.ts
  documents.ts
```

### 4.4 Schema 任务

`documents` 需要先落这些字段：

- `userId`
- `knowledgeBaseId`
- `sourceType`
- `contentFormat`
- `originalName`
- `extension`
- `fileType`
- `mimeType`
- `size`
- `status`

可在本阶段一起加上的实用字段：

- `chunkCount`
- `processedAt`
- `storageKey`
- `errorMessage`
- `splitterType`
- `splitConfig`

### 4.5 DTO 任务

至少补这些 DTO：

- `ListDocumentsQueryDto`
- `UploadDocumentDto`
- `CreateEditorDocumentDto`
- `UpdateDocumentStatusDto`

其中：

- `UploadDocumentDto` 只放业务字段，不承载文件二进制
- `CreateEditorDocumentDto` 负责承接 Markdown 编辑器的纯文本内容

### 4.6 Service 任务

`DocumentsService` 至少要拆出这些方法：

- `createUploadDocument(...)`
- `createEditorDocument(...)`
- `findAll(...)`
- `findOne(...)`
- `remove(...)`
- `updateStatus(...)`

Service 里必须做的校验：

- `knowledgeBaseId` 必须归属于当前用户
- 删除文档时必须校验文档归属
- 查询列表必须按 `userId` 限定

### 4.7 文件上传和编辑器内容如何区分

上传文件：

- `sourceType = upload`
- `contentFormat` 根据最终可切分内容决定
  - `.md` -> `markdown`
  - `pdf/doc/docx` -> `plain_text`

编辑器内容：

- `sourceType = editor`
- `contentFormat = markdown`

### 4.8 Storage 任务

当前 `StorageService` 已存在，但需要收口：

- 统一对象存储字段命名
- `documents.storageKey` 统一作为对象存储路径字段
- 上传删除逻辑都围绕 `storageKey` 展开

如果编辑器内容也要保留原文，可选两种方案：

1. 直接把原文存 MongoDB 之外的对象存储
2. 第一阶段只存数据库主记录和 chunk，不额外保存原文对象

最小闭环推荐：

- 上传文件写对象存储
- 编辑器纯文本先不强制写对象存储

### 4.9 测试任务

本阶段至少补这些测试：

- 创建上传文档时，`sourceType/contentFormat/fileType` 是否正确
- 创建编辑器文档时，`sourceType=editor`、`contentFormat=markdown` 是否正确
- `GET /documents` 是否只返回当前用户数据
- `DELETE /documents/:id` 是否正确删除主记录

### 4.10 验收标准

- 用户能查询自己的文档列表
- 上传 `md/pdf/docx` 后能生成正确主记录
- 编辑器 Markdown 文本能生成文档主记录
- 文档记录具备后续切分所需的最小字段

## 5. 阶段二：`document_chunks`

### 5.1 目标

把 `documents` 转成真正可检索的 chunk 数据。

### 5.2 本阶段交付

- `document_chunks` schema
- chunk 重建逻辑
- 文档切分任务服务
- 文档状态从 `pending/processing` 到 `ready/failed`

### 5.3 需要新增或改动的代码

推荐新增：

```text
src/documents/schemas/document_chunk.schema.ts
src/documents/documents.types.ts
```

推荐在 `DocumentsService` 中补：

- `splitDocument(...)`
- `splitUploadDocument(...)`
- `splitEditorDocument(...)`
- `rebuildChunks(...)`
- `deleteChunksByDocumentId(...)`

### 5.4 Schema 任务

`document_chunks` 至少落这些字段：

- `userId`
- `knowledgeBaseId`
- `documentId`
- `sequence`
- `content`
- `charCount`
- `startIndex`
- `metadata`

推荐同步落这些实用字段：

- `endIndex`
- `tokenCount`

第一阶段先不要把 `embedding` 做成必填。

### 5.5 LangChain 切分实现任务

本阶段核心不是数据库，而是“统一切分入口”。

你需要明确这几类输入的处理方式：

- `pdf` -> `PDFLoader`
- `docx` -> `DocxLoader`
- `doc` -> `DocxLoader({ type: 'doc' })`
- `md` 文件 -> `TextLoader`
- Markdown 纯文本 -> 直接构造 LangChain `Document`

然后统一走：

```text
LangChain Document[]
  -> RecursiveCharacterTextSplitter
  -> splitDocuments(...)
  -> 映射为 document_chunks
```

### 5.6 `metadata` 映射任务

`metadata` 至少统一成这套结构：

- `sourceType`
- `contentFormat`
- `page`
- `headingPath`
- `blockType`

建议：

- `page` 主要给 PDF
- `headingPath` 主要给 Markdown 文件和编辑器内容
- `blockType` 第一阶段可选，后续再细化

### 5.7 `documents` 状态维护任务

切分开始前：

- `status = processing`

切分成功后：

- `status = ready`
- `chunkCount = 实际数量`
- `processedAt = 当前时间`

切分失败后：

- `status = failed`
- `errorMessage = 错误摘要`

### 5.8 测试任务

至少补这些测试：

- 同一文档切分后 `sequence` 是否连续
- `startIndex` 是否正确生成
- PDF chunk 是否能保留 `page`
- Markdown chunk 是否能保留 `headingPath`
- 重建 chunk 时是否会先清理旧 chunk

### 5.9 验收标准

- 上传或编辑器文档都能生成 chunk
- chunk 具备最小来源定位能力
- 文档状态能正确更新
- 删除文档时能清理关联 chunk

## 6. 阶段三：`chat_sessions / chat_messages`

### 6.1 目标

把聊天过程稳定持久化，为后续 `chat/ask` 和 `chat/stream` 提供基础。

### 6.2 本阶段交付

- `ChatModule`
- `chat_sessions` schema
- `chat_messages` schema
- 会话列表接口
- 历史消息接口

推荐接口：

- `GET /chat/sessions`
- `GET /chat/sessions/:id/messages`

### 6.3 需要新增的代码

```text
src/chat/
  dto/
  schemas/
  chat.controller.ts
  chat.module.ts
  chat.service.ts
  chat.ts
```

### 6.4 `chat_sessions` 任务

`chat_sessions` 最少字段：

- `userId`
- `knowledgeBaseId`
- `title`
- `messageCount`
- `lastMessageAt`

推荐一起加上：

- `lastMessagePreview`

### 6.5 `chat_messages` 任务

`chat_messages` 最少字段：

- `userId`
- `knowledgeBaseId`
- `sessionId`
- `messageType`
- `content`
- `sequence`

推荐一起加上：

- `messageId`
- `name`
- `sources`
- `responseMetadata`
- `usageMetadata`
- `toolCalls`
- `toolCallId`
- `toolName`
- `artifact`

### 6.6 LangChain 消息类型映射任务

数据库层统一使用：

- `system`
- `human`
- `ai`
- `tool`

不要继续保留：

- `user`
- `assistant`

否则后面 LangChain 映射会越来越乱。

### 6.7 Service 任务

`ChatService` 至少要具备这些基础方法：

- `createSession(...)`
- `findSessions(...)`
- `findMessagesBySessionId(...)`
- `appendHumanMessage(...)`
- `appendAiMessage(...)`
- `appendToolMessage(...)`

同时维护：

- `messageCount`
- `lastMessageAt`
- `lastMessagePreview`

### 6.8 测试任务

- 会话只能访问当前用户自己的知识库数据
- 消息顺序 `sequence` 是否正确
- 创建首条问题时是否能自动生成会话标题
- 会话摘要字段是否随消息写入正确更新

### 6.9 验收标准

- 会话列表可查
- 历史消息可回放
- 消息类型已经对齐 LangChain
- 后续 `chat/ask` 可以直接复用这些持久化能力

## 7. 阶段四：`chat/ask`

### 7.1 目标

用 LangChain 跑通非流式 JSON 问答。

### 7.2 本阶段交付

- `POST /chat/ask`
- 检索当前知识库相关 chunk
- 调用 LangChain chat model
- 返回 `answer + sources`
- 落库 `human` 消息和 `ai` 消息

### 7.3 需要新增的 DTO

- `AskQuestionDto`

字段最少包括：

- `knowledgeBaseId`
- `sessionId`
- `question`

### 7.4 Service 任务

`ChatService.ask(...)` 至少拆成这些步骤：

1. 校验当前用户是否拥有该 `knowledgeBaseId`
2. 创建或读取 `chat_session`
3. 保存一条 `human` 消息
4. 查询当前知识库下的 `document_chunks`
5. 选出 topK 上下文
6. 组装 LangChain messages
7. 调用 `ChatOpenAI.invoke(...)`
8. 生成 `sources`
9. 保存 `ai` 消息
10. 返回响应结构

### 7.5 LangChain 对接任务

你需要引入并封装：

```ts
import { ChatOpenAI } from "@langchain/openai";
import { SystemMessage, HumanMessage, AIMessage, ToolMessage } from "@langchain/core/messages";
```

同时把数据库消息映射为运行时消息对象。

### 7.6 检索任务

最小闭环下先不要急着上向量库。

你可以先做两种方案中的一种：

1. 先用 MongoDB 普通文本检索或关键词匹配验证主链路
2. 如果你已经准备好 embedding，再接向量检索

但无论哪种方式，都必须：

- 限定 `userId + knowledgeBaseId`
- 返回结构化 `sources`

### 7.7 `sources` 生成任务

每条 source 至少生成：

- `knowledgeBaseId`
- `documentId`
- `documentName`
- `chunkId`
- `chunkSequence`
- `snippet`

有条件再加：

- `page`
- `headingPath`
- `startIndex`
- `score`

### 7.8 测试任务

- 问答只能检索当前知识库
- `sources` 是否随 `ai` 消息稳定落库
- 历史会话继续提问时是否沿用同一 `sessionId`
- `usageMetadata`、`responseMetadata` 是否能正确保存

### 7.9 验收标准

- `POST /chat/ask` 能稳定返回答案
- 前端能直接显示引用来源
- 聊天记录能在历史会话中回放

## 8. 阶段五：`chat/stream`

### 8.1 目标

在阶段四稳定之后，再补上 SSE 流式问答。

### 8.2 本阶段交付

- `POST /chat/stream`
- 基于 LangChain 流式输出逐步推送内容
- 结束后统一保存完整 `ai` 消息

### 8.3 关键约束

不要：

- 每收到一个 token / chunk 就写一次数据库

应该：

1. LangChain 流式输出时持续向前端转发
2. 服务层在内存中聚合完整文本
3. 流结束后一次性保存 `ai` 消息
4. 同时保存 `sources`

### 8.4 需要新增的 DTO

- `StreamChatDto`

字段与 `AskQuestionDto` 基本一致：

- `knowledgeBaseId`
- `sessionId`
- `question`

### 8.5 Controller 任务

控制器需要明确：

- SSE 响应头
- 错误事件
- 完成事件

推荐至少定义这些事件类型：

- `message`
- `error`
- `done`

### 8.6 Service 任务

`ChatService.stream(...)` 至少做这些事：

1. 校验知识库归属
2. 保存 `human` 消息
3. 检索上下文
4. 调用 LangChain 流式模型
5. 逐步向前端输出
6. 聚合完整 `ai` 内容
7. 流结束后保存 `ai` 消息与 `sources`

### 8.7 测试任务

- SSE 是否能持续输出
- 出错时是否能正确中断
- 完成后是否会落一条完整 `ai` 消息
- 是否不会产生多条碎片化消息记录

### 8.8 验收标准

- 聊天页能实时看到回答增长
- 数据库中只保留一条完整 `ai` 消息
- 流式和非流式返回的消息结构保持一致

## 9. 推荐的代码提交顺序

如果你准备按 PR 或 commit 分阶段推进，推荐这样拆：

1. `refactor(documnets): rename to documents and align document schema`
2. `feat(documents): add upload/editor document records and list api`
3. `feat(document-chunks): add langchain split pipeline and chunk persistence`
4. `feat(chat): add sessions and messages persistence`
5. `feat(chat): add json ask api with sources persistence`
6. `feat(chat): add sse stream api with final ai message persistence`

这样拆的好处是：

- 每个阶段边界清楚
- 回归测试范围可控
- 即使中途停下，仓库状态也仍然可运行

---

## 10. 每阶段结束后都要复核什么

每推进完一个阶段，都建议固定复核这几件事：

### 10.1 权限边界

- 是否所有查询都带了 `userId`
- 是否所有子资源都严格归属于 `knowledgeBaseId`

### 10.2 数据冗余

- 是否把本应放在 `documents` 的配置塞进了 `document_chunks`
- 是否把临时运行态错误地直接落到了长期表结构里

### 10.3 LangChain 映射

- 是否把第三方对象原样塞进数据库
- 是否已经统一为项目自己的稳定字段

### 10.4 删除策略

- 删除文档时是否清理 chunk
- 删除知识库时是否准备好级联删除后续子资源
- 历史 `sources` 快照是否不会因为文档删除而直接失效
