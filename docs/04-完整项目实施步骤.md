## 1. 4 周主线学习路线

这一节直接承接原版文档第 19 章的学习节奏，并补充当前仓库的实现进度。

### 第 1 周：把项目骨架和配置跑起来

目标：

- 理解 Nest 的模块、控制器、服务
- 跑通 Swagger
- 连通 MongoDB
- 看懂当前项目的目录职责

交付结果：

- 项目启动成功
- Swagger 可以打开
- MongoDB 连接成功
- 用户和知识库基础模块能正常工作

### 第 2 周：做完整认证系统

目标：

- 注册
- 登录
- accessToken 鉴权
- refreshToken 刷新
- logout 失效

交付结果：

- 可以拿着 accessToken 访问受保护接口
- accessToken 过期后可以刷新

### 第 3 周：完成知识库与文档主链路

目标：

- 创建知识库
- 新建文档模块
- 做用户级文档列表
- 设计文档状态流转和 chunk 模型

交付结果：

- 当前用户所有知识库下的文档可统一查询
- 文档能按文件类型区分
- 文档 schema 和 chunk schema 有最小可用字段定义

### 第 4 周：完成 RAG 问答与 SSE

目标：

- 根据问题检索相关 chunk
- 基于检索结果生成回答
- 返回引用来源
- 支持 JSON 和 SSE 两种输出

交付结果：

- `POST /chat/ask` 正常返回答案
- `POST /chat/stream` 能边生成边返回

## 2. 环境变量设计

这部分直接承接原版文档第 18 章的结构。原文有一份完整 `.env.example`，当前版本保留它的分组思路，但把“当前必需”和“后续预留”拆开讲。

### 2.1 当前仓库必需环境变量

当前代码真正依赖的变量只有这几项：

| 变量 | 作用 | 当前是否必需 |
| --- | --- | --- |
| `NODE_ENV` | 运行环境 | 是 |
| `PORT` | 服务端口 | 否，缺省可用默认值 |
| `MONGODB_URI` | MongoDB 连接串 | 是 |
| `JWT_ACCESS_SECRET` | accessToken 签名密钥 | 否，缺省有兜底值 |
| `JWT_ACCESS_EXPIRESIN` | accessToken 有效期 | 否 |
| `JWT_REFRESH_SECRET` | refreshToken 签名密钥 | 否，缺省有兜底值 |
| `JWT_REFRESH_EXPIRESIN` | refreshToken 有效期 | 否 |

设计原则：

- 生产环境必须显式配置 JWT 密钥，不依赖默认值
- `MONGODB_URI` 是当前唯一真正不能缺的核心变量
- 当前项目不要求对象存储、模型、向量检索等变量

### 2.2 完整项目 `.env.example` 参考

下面这份结构保留自原版文档，但这里明确说明：

- 它是完整项目蓝图的环境变量设计
- 不是当前仓库部署的最低要求

```env
NODE_ENV=development
PORT=3000
APP_NAME=knowledge-base-api

MONGODB_URI=mongodb+srv://<username>:<password>@cluster0.xxxxx.mongodb.net/?retryWrites=true&w=majority&appName=Cluster0
MONGODB_DB_NAME=knowledge_base_app
MONGODB_VECTOR_INDEX_NAME=kb_chunk_vector_index

JWT_ACCESS_SECRET=replace-with-a-very-long-random-string
JWT_ACCESS_EXPIRESIN=15m
JWT_REFRESH_SECRET=replace-with-another-very-long-random-string
JWT_REFRESH_EXPIRESIN=30d
BCRYPT_SALT_ROUNDS=10

R2_ACCOUNT_ID=your-cloudflare-account-id
R2_ACCESS_KEY_ID=your-r2-access-key-id
R2_SECRET_ACCESS_KEY=your-r2-secret-access-key
R2_BUCKET_NAME=knowledge-base-files
R2_REGION=auto

OPENAI_API_KEY=your-openai-compatible-api-key
OPENAI_BASE_URL=https://your-compatible-base-url.com/v1
OPENAI_CHAT_MODEL=gpt-4.1-mini
OPENAI_EMBEDDING_MODEL=text-embedding-3-small
OPENAI_EMBEDDING_DIMENSIONS=1536
OPENAI_STREAM_USAGE=false

RAG_CHUNK_SIZE=1000
RAG_CHUNK_OVERLAP=200
RAG_TOP_K=5
UPLOAD_MAX_FILE_SIZE_MB=10
```

### 2.3 建议按领域分组

推荐按这几组管理：

#### 应用基础

- `NODE_ENV`
- `PORT`
- `APP_NAME`

#### 数据库

- `MONGODB_URI`
- `MONGODB_DB_NAME`
- `MONGODB_VECTOR_INDEX_NAME`

#### 鉴权

- `JWT_ACCESS_SECRET`
- `JWT_ACCESS_EXPIRESIN`
- `JWT_REFRESH_SECRET`
- `JWT_REFRESH_EXPIRESIN`
- `BCRYPT_SALT_ROUNDS`

#### 对象存储

- `R2_ACCOUNT_ID`
- `R2_ACCESS_KEY_ID`
- `R2_SECRET_ACCESS_KEY`
- `R2_BUCKET_NAME`
- `R2_REGION`

#### 模型与 RAG

- `OPENAI_API_KEY`
- `OPENAI_BASE_URL`
- `OPENAI_CHAT_MODEL`
- `OPENAI_EMBEDDING_MODEL`
- `OPENAI_EMBEDDING_DIMENSIONS`
- `OPENAI_STREAM_USAGE`
- `RAG_CHUNK_SIZE`
- `RAG_CHUNK_OVERLAP`
- `RAG_TOP_K`

### 2.4 每个变量的大意

| 变量 | 说明 |
| --- | --- |
| `MONGODB_URI` | MongoDB 连接串 |
| `MONGODB_DB_NAME` | 业务数据库名 |
| `MONGODB_VECTOR_INDEX_NAME` | 向量索引名称 |
| `JWT_ACCESS_SECRET` | accessToken 签名密钥 |
| `JWT_ACCESS_EXPIRESIN` | accessToken 默认有效期 |
| `JWT_REFRESH_SECRET` | refreshToken 签名密钥 |
| `JWT_REFRESH_EXPIRESIN` | refreshToken 默认有效期 |
| `R2_BUCKET_NAME` | 存原始文件的 bucket |
| `OPENAI_BASE_URL` | OpenAI 兼容接口地址 |
| `OPENAI_CHAT_MODEL` | 聊天模型名称 |
| `OPENAI_EMBEDDING_MODEL` | Embedding 模型名称 |
| `OPENAI_EMBEDDING_DIMENSIONS` | 向量维度，必须和索引一致 |

### 2.5 环境变量设计最容易踩的坑

- 把当前项目不需要的变量误当成部署必需项
- 布尔值直接按字符串使用，导致 `false` 被当真
- Embedding 维度和向量索引维度不一致
- 把密钥散落在代码里，后续很难统一调整

## 3. 各个阶段的讲解说明

下面这些阶段说明，是从原版文档第 20 到 29 章提炼出来的。原文的优势在于“为什么要做”讲得足够细，所以这里保留那种教学节奏，同时对和当前仓库不一致的地方做修正。

每个阶段都按同一个模板来理解：

- 这一阶段在解决什么问题
- 为什么要先做它
- 你会接触哪些模块和文件
- 这一阶段完成后能得到什么
- 常见错误是什么

额外加一条：

- 这一阶段最少要落哪些字段，哪些字段可以先不做

### 3.1 阶段 1：环境准备

当前状态：部分已实现

这一阶段在解决什么问题：

- 把项目运行环境先搭起来

为什么要先做它：

- 没有环境，后面所有开发都无法验证

会接触哪些内容：

- Node.js
- pnpm
- MongoDB
- `.env`

完成后能得到什么：

- 本地能启动服务
- 本地能连接数据库

常见错误：

- 连接串里的密码有特殊字符却没编码
- 误以为当前仓库已经强依赖 R2 和模型
- `.env` 写了，但程序并没有真正读取到

### 3.2 阶段 2：Nest 项目初始化

当前状态：已实现

这一阶段在解决什么问题：

- 搭好后端工程骨架

为什么要先做它：

- 如果一开始工程结构就乱，后面模块越多越难维护

会接触哪些内容：

- `AppModule`
- Swagger
- 全局 ValidationPipe
- `common` 模块能力

完成后能得到什么：

- 一个能持续扩展的后端基础骨架

常见错误：

- Swagger 装了依赖但没有 `setup`
- DTO 写了校验却没启用全局 `ValidationPipe`

### 3.3 阶段 3：配置管理

当前状态：已实现基础版

这一阶段在解决什么问题：

- 让配置集中管理，而不是散落在代码里

为什么要先做它：

- 数据库、JWT、模型、对象存储后面都会依赖配置

会接触哪些内容：

- `ConfigModule`
- `ConfigService`
- `.env`

完成后能得到什么：

- 配置读取路径统一

常见错误：

- 数字和布尔值没做转换
- 当前值依赖默认值，到了生产环境才发现密钥没配

### 3.4 阶段 4：MongoDB 连接与基础 Schema

当前状态：已实现基础版

这一阶段在解决什么问题：

- 把数据真正稳定地存起来

为什么要先做它：

- 没有 schema 和索引，数据结构很快会失控

会接触哪些内容：

- Mongoose 连接
- `users` schema
- `knowledge_bases` schema

完成后能得到什么：

- 用户和知识库主记录可持久化

常见错误：

- 忘记加索引
- 忘记统一 timestamps 和序列化规则

### 3.5 阶段 5：用户认证

当前状态：已实现

这一阶段在解决什么问题：

- 明确“当前请求是谁”

为什么要先做它：

- 没有身份体系，知识库、文档、会话都无法按用户隔离

会接触哪些内容：

- `users.controller.ts`
- `users.service.ts`
- JWT
- Guard

完成后能得到什么：

- 完整登录链路
- 受保护接口可以识别当前用户

常见错误：

- refreshToken 只存明文
- accessToken 和 refreshToken 没有清楚区分用途
- 登录成功了，但后续接口读不到 `req.user`

### 3.6 阶段 6：知识库管理

当前状态：已实现主记录层

这一阶段在解决什么问题：

- 给用户一个内容容器

为什么要先于文档做：

- 文档和会话都必须归属于知识库

会接触哪些内容：

- `knowledge_bases.controller.ts`
- `knowledge_bases.service.ts`
- `knowledge_base.schema.ts`

完成后能得到什么：

- 用户可以管理自己的知识库

常见错误：

- 只做名字和描述，不提前考虑汇总字段
- 忘记按 `userId` 限制查询

这一阶段最少要落的字段：

- `userId`
- `name`
- `description`
- `documentCount`
- `chunkCount`
- `sessionCount`

可以后补的字段：

- `lastIndexedAt`

### 3.7 阶段 7：文档模块

当前状态：规划中

这一阶段在解决什么问题：

- 把“知识库里有什么内容”管理起来

为什么这一阶段优先级高：

- 前端已经有明确页面需要用户级文档列表

会接触哪些内容：

- `documents` 集合
- 用户级文档列表接口
- 文件类型映射

完成后能得到什么：

- 当前用户所有知识库下的文档统一可查

常见错误：

- 只存扩展名，不做展示级 `fileType`
- 把文档列表强行挂在知识库详情里

这一阶段最少要落的字段：

- `userId`
- `knowledgeBaseId`
- `originalName`
- `extension`
- `fileType`
- `mimeType`
- `size`
- `status`

可以后补的字段：

- `statusMessage`
- `chunkCount`
- `processedAt`
- `storageKey`
- `errorMessage`

### 3.8 阶段 8：文档切片与 chunk

当前状态：规划中

这一阶段在解决什么问题：

- 让文档变成可检索的数据

为什么这一步关键：

- 模型检索的是 chunk，不是整份文档

会接触哪些内容：

- `document_chunks`
- chunk 顺序
- chunk 统计

完成后能得到什么：

- 文档可以被检索

常见错误：

- 把所有 chunk 直接嵌进文档里
- 没有稳定顺序字段
- 删文档时忘记删 chunk

这一阶段最少要落的字段：

- `userId`
- `knowledgeBaseId`
- `documentId`
- `sequence`
- `content`
- `charCount`

可以后补的字段：

- `tokenCount`
- `embedding`
- `metadata`

### 3.9 阶段 9：会话与消息

当前状态：规划中

这一阶段在解决什么问题：

- 把问答过程持久化

为什么不要跳过这一步：

- 没有会话和消息模型，聊天页无法成立

会接触哪些内容：

- `chat_sessions`
- `chat_messages`
- `role`

完成后能得到什么：

- 会话列表
- 消息流

常见错误：

- 只存最终答案，不存完整消息流
- 会话和消息混成一张表

这一阶段最少要落的字段：

`chat_sessions`：

- `userId`
- `knowledgeBaseId`
- `title`
- `messageCount`
- `lastMessageAt`

`chat_messages`：

- `userId`
- `knowledgeBaseId`
- `sessionId`
- `role`
- `content`
- `sequence`

可以后补的字段：

- `sources`
- `model`
- `usage`
- `finishReason`

### 3.10 阶段 10：RAG 问答

当前状态：规划中

这一阶段在解决什么问题：

- 让系统基于知识库回答，而不是泛聊天

为什么这一阶段是项目灵魂：

- 前面所有数据准备，最终都在为“基于知识库回答”服务

会接触哪些内容：

- 检索
- Prompt
- 回答生成
- 引用来源

完成后能得到什么：

- 回答可追溯
- 回答和知识库强绑定

常见错误：

- 检索为空时仍让模型自由发挥
- 没有把引用来源作为结构化结果返回

这一阶段最少要落的结构：

- 输入：`knowledgeBaseId`、`question`
- 输出：`answer`、`sources`
- 存储：至少写入一条用户消息和一条助手消息

### 3.11 阶段 11：SSE 流式问答

当前状态：规划中

这一阶段在解决什么问题：

- 提供更接近聊天产品的实时交互体验

为什么它放在最后：

- SSE 在控制器、服务、前端联调三端都更复杂

会接触哪些内容：

- 流式响应
- token 推送
- 最终落库

完成后能得到什么：

- 实时输出
- 流式与非流式共存

常见错误：

- 每收到一个 token 就写数据库
- 忘记发送 `done` 事件

## 4. 每阶段完成后的验收重点

### 4.1 用户认证

- 能注册
- 能登录
- 能刷新令牌
- 能退出登录

### 4.2 知识库

- 只能看到自己的知识库
- 分页正确
- 汇总字段初始值正确

### 4.3 文档

- 能拿到当前用户所有知识库下的文档列表
- `fileType` 映射正确
- 状态展示正确

### 4.4 问答

- 仅在当前知识库内检索
- 消息能按会话顺序保存
- SSE 能边生成边返回
