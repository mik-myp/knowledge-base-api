## 2. 项目目标

这个项目是一个面向个人用户的知识库问答后端，完整目标包括：

1. 用户注册、登录、刷新令牌、退出登录。
2. 用户创建多个知识库。
3. 知识库下管理多份文档，并区分文件类型。
4. 文档解析为多个 chunk。
5. 通过向量检索为问答提供上下文。
6. 在知识库维度保存问答会话与消息。
7. 同时支持普通 JSON 问答与 SSE 流式问答。

## 3. 当前仓库进度

当前代码仓库已经落地的能力：

- `users` 模块：注册、登录、刷新、退出、当前用户信息
- `knowledge-bases` 模块：知识库增删改查
- MongoDB 连接、统一响应拦截器、JWT 鉴权、Swagger

当前文档中已经定稿、但代码尚未落地的能力：

- `documents`
- `document_chunks`
- `chat_sessions`
- `chat_messages`
- 文档上传、切片、Embedding、RAG、SSE

换句话说，当前仓库是“完整项目的前半段骨架”，文档会继续覆盖完整蓝图，但会明确标记哪些是“当前已实现”，哪些是“后续实现计划”。

## 6. 当前与目标的边界

为了避免文档与代码脱节，后续所有分册遵循这条规则：

- `当前已实现`：当前仓库中已经存在、可以运行、可以测试的内容
- `规划中`：已经定稿的数据模型或接口方向，但代码还没落地
- `教程蓝图`：为了从 0 到 1 完整实现项目而保留的教学内容
- `产品文档`：解释项目价值、用户场景、交互逻辑、业务流转
- `规范文档`：解释项目应该如何写、如何拆、如何保持一致性

## 7. 推荐下一步

如果你要继续推进代码实现，建议顺序是：

1. 先完成 `documents` 模块与用户文档列表接口。
2. 再完成 `document_chunks` 与文档状态流转。
3. 然后补 `chat_sessions` / `chat_messages`。
4. 最后接入 RAG 检索、引用来源与 SSE。

这样能保持接口、数据模型和前端需求同步前进，而不会一下子把实现面铺得太大。
