## 3. 当前响应结构

当前项目启用了全局响应拦截器，最终 HTTP 响应会统一为：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {},
  "timestamp": "2026-03-20T00:00:00.000Z",
  "path": "/knowledge-bases"
}
```

其中：

- `users` 控制器会先通过 `ResponseUtil` 返回标准结构
- `knowledge-bases` 控制器返回业务数据后，再由全局拦截器包一层标准结构

因此阅读当前代码时，要区分：

- service 的业务返回结构
- 最终 HTTP 响应结构

## 4. 从页面到接口的调用链

这部分是给前端背景读者补的。

### 4.1 登录页

前端动作：

- 用户输入邮箱和密码

后端调用：

- `POST /users/login`

后端处理链：

```text
LoginDto
  -> UsersController.login()
  -> UsersService.login()
  -> users 集合校验账号
  -> 返回 user + accessToken + refreshToken
```

### 4.2 知识库列表页

前端动作：

- 获取当前用户知识库列表

后端调用：

- `GET /knowledge-bases?page=1&pageSize=10`

后端处理链：

```text
ListKnowledgeBasesQueryDto
  -> KnowledgeBasesController.findAll()
  -> KnowledgeBasesService.findAll()
  -> knowledge_bases 集合分页查询
  -> 返回 dataList + total
```

### 4.3 创建知识库弹窗

前端动作：

- 提交知识库名称和描述

后端调用：

- `POST /knowledge-bases`

后端处理链：

```text
CreateKnowledgeBaseDto
  -> KnowledgeBasesController.create()
  -> KnowledgeBasesService.create()
  -> knowledge_bases 集合写入主记录
  -> 初始化汇总字段
```

## 5. API 设计与 DTO 设计

### 5.1 DTO 设计原则

这一段直接承接原版文档第 17 章最前面的规则。

先记住 5 条规则：

1. DTO 只负责 HTTP 边界校验
2. service 内部传递的数据优先用 `type`
3. `:id` 这类单一路径参数优先交给 `ParseObjectIdPipe`
4. 列表接口先从最小分页 DTO 开始，不急着引入复杂搜索
5. 当前项目优先选择显式清晰的 DTO 类，而不是过早抽象

### 5.2 当前 API 设计

#### 用户接口

| 方法 | 路径 | 说明 | 鉴权 |
| --- | --- | --- | --- |
| `POST` | `/users/register` | 用户注册 | 否 |
| `POST` | `/users/login` | 用户登录 | 否 |
| `POST` | `/users/refresh` | 刷新令牌 | 否 |
| `POST` | `/users/logout` | 退出登录 | 是 |
| `GET` | `/users/me` | 获取当前用户 | 是 |

#### 知识库接口

| 方法 | 路径 | 说明 | 鉴权 |
| --- | --- | --- | --- |
| `POST` | `/knowledge-bases` | 创建知识库主记录 | 是 |
| `GET` | `/knowledge-bases` | 获取知识库列表 | 是 |
| `GET` | `/knowledge-bases/:id` | 获取知识库详情 | 是 |
| `PATCH` | `/knowledge-bases/:id` | 更新知识库 | 是 |
| `DELETE` | `/knowledge-bases/:id` | 删除知识库 | 是 |

设计特点：

- 认证能力全部归并在 `users` 模块
- 当前知识库接口只负责主记录和汇总字段
- 当前还没有文档和问答接口

### 5.3 当前 DTO 设计

#### 用户模块 DTO

`RegisterDto`

- 路由：`POST /users/register`

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `email` | string | 是 | 用户邮箱 |
| `password` | string | 是 | 登录密码，8-32 位 |
| `username` | string | 是 | 用户昵称，2-30 位 |

`LoginDto`

- 路由：`POST /users/login`

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `email` | string | 是 | 用户邮箱 |
| `password` | string | 是 | 登录密码 |

`RefreshTokenDto`

- 路由：`POST /users/refresh`

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `refreshToken` | string | 是 | 用于刷新 accessToken 的 refreshToken |

#### 知识库模块 DTO

`CreateKnowledgeBaseDto`

- 路由：`POST /knowledge-bases`

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `name` | string | 是 | 知识库名称 |
| `description` | string | 否 | 知识库描述 |

`UpdateKnowledgeBaseDto`

- 路由：`PATCH /knowledge-bases/:id`

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `name` | string | 否 | 更新后的知识库名称 |
| `description` | string | 否 | 更新后的知识库描述 |

`ListKnowledgeBasesQueryDto`

- 路由：`GET /knowledge-bases`

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `page` | number | 否 | 页码 |
| `pageSize` | number | 否 | 每页数量，最大 50 |

### 5.4 哪些接口当前不需要 DTO

当前这些接口没有单独 DTO 是合理的：

- `GET /users/me`
- `POST /users/logout`
- `GET /knowledge-bases/:id`
- `DELETE /knowledge-bases/:id`

原因：

- 没有请求体
- 只有单个路径参数
- 路径参数校验已经由 `ParseObjectIdPipe` 承接

### 5.5 当前哪些类型不是 DTO

当前这些更适合用 `type` 而不是 DTO：

- `AccessTokenPayload`
- `RefreshTokenPayload`
- `UserProfile`
- `TokenPairResult`
- `KnowledgeBaseRecord`
- `KnowledgeBaseListResult`
- `UserRequest`

## 6. 当前代码阅读顺序

如果你要理解当前代码，建议这样看：

1. 先看 controller，知道有哪些接口
2. 再看 DTO，知道接口收什么参数
3. 再看 service，知道业务怎么走
4. 再看 schema，知道数据最终怎么存

这比一上来直接钻进 schema 或 service 更容易建立整体感。

### 6.1 当前代码如何作为后续模块模板

虽然当前仓库还没有 `documents`、`chat`、`rag` 模块，但现有 `users` 和 `knowledge_bases` 已经足够当成模板来参考：

- 想看一个已落地 schema 怎么写，看 `users` 和 `knowledge_bases`
- 想看一个模块怎么注册 `MongooseModule.forFeature(...)`，看当前两个 module
- 想看 service 怎么统一返回对象结构，看 `knowledge_bases.service.ts`
- 想看 DTO 如何只承担 HTTP 入参校验，看现有 `dto/` 目录

也就是说，后续模块不是从零发明写法，而是沿用当前仓库已经存在的风格。

## 7. 当前 Swagger 边界

当前 Swagger 应该只展示两类真实接口：

- `users`
- `knowledge-bases`

当前知识库接口的语义边界是：

- 只管理知识库主数据
- 只返回知识库汇总字段
- 不承诺已经实现文档、会话、消息等子资源
