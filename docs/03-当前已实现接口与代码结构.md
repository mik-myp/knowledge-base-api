## 3. 当前文档模块已经做完的事情

### 3.1 文档主表

`documents` 已经是收口后的最小主表，保留：

- `userId`
- `knowledgeBaseId`
- `sourceType`
- `originalName`
- `storageKey`
- `content`
- `extension`
- `mimeType`
- `size`

### 3.2 chunk 业务表

`document_chunks` 已经落地，不再只是文档设计稿。

当前 schema 字段：

- `userId`
- `knowledgeBaseId`
- `documentId`
- `sequence`
- `content`
- `page`
- `startIndex`
- `endIndex`

### 3.3 向量表

后端已经通过 `DocumentIndexingService` 直接写入 Mongo 原生 collection：

- `document_chunk_vectors`

当前写入字段：

- `userId`
- `knowledgeBaseId`
- `documentId`
- `documentName`
- `sequence`
- `text`
- `embedding`
- `page`
- `startIndex`
- `endIndex`

### 3.4 文本提取与切片

当前支持的上传扩展名：

- `md`
- `markdown`
- `txt`
- `pdf`
- `doc`
- `docx`

当前行为：

- `md/markdown/txt` 直接读取 UTF-8 文本
- `docx` 用 `mammoth` 提取纯文本
- `pdf` 用 `pdf-parse` 逐页取文本
- `doc` 当前直接拒绝解析，提示先转 `docx`
- 切片器使用 `RecursiveCharacterTextSplitter`

### 3.5 定位字段

当前索引服务已经支持来源定位：

- `md/txt/docx` 手动计算 `startIndex` / `endIndex`
- `pdf` 保留 `page`
- `pdf` 也会在页内继续计算 `startIndex` / `endIndex`

### 3.6 向量化限制

当前 embedding 批大小已经在服务层做了限制：

- `OPENAI_EMBEDDING_BATCH_SIZE` 最大只取到 `10`

这是因为当前接入的 embedding 服务要求单批不能大于 `10`。

### 3.7 上传失败整体回滚

`POST /documents/upload` 和 `POST /documents/editor` 当前都已经是“事务式效果”：

- 先建文档
- 再切片
- 再写 `document_chunks`
- 再写 `document_chunk_vectors`
- 任一环节失败，回滚已创建的文档记录、chunk、向量、对象存储文件

所以当前设计目标已经不是“允许部分成功”，而是“要么整份文档成功，要么整份文档失败”。

### 3.8 批量删除

`DELETE /documents/all` 已经存在，对应：

- `DocumentsService.removeByDocumentIds`

当前行为：

- 删除 `documents`
- 删除对象存储文件
- 删除 `document_chunks`
- 删除 `document_chunk_vectors`
