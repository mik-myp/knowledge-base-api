## 1. 项目在解决什么问题

一句话版：

> 用户上传自己的资料，系统先从资料里检索相关内容，再基于检索结果回答问题，而不是脱离知识库胡乱作答。

这个项目的价值不只是“做一个聊天接口”，而是把下面几件事串起来：

- 用户体系
- 知识库管理
- 文档管理
- 文档切片与检索
- 问答会话持久化
- 面向真实产品的接口组织

如果你想先从产品视角理解这件事，先看 [00-项目产品与业务总览](./00-项目产品与业务总览.md)。

---

## 2. 项目产品定位

当前产品定位是：

- 面向个人用户的知识库问答后端
- 先服务“单用户、多知识库、文档问答”的核心主线
- 暂不扩展到团队协作、权限体系、多租户后台

这意味着文档和代码都优先围绕一条主线展开：

```text
登录
  -> 创建知识库
  -> 管理文档
  -> 文档处理为可检索数据
  -> 在知识库中发问
  -> 保存会话与消息
```

---

## 3. 当前仓库与最终目标

### 3.1 当前仓库已实现

- 用户注册、登录、刷新令牌、退出登录、获取当前用户
- 知识库增删改查
- MongoDB 接入
- JWT 鉴权
- Swagger 文档
- 统一响应结构

### 3.2 完整目标蓝图

- 一个用户有多个知识库
- 一个知识库有多个文档
- 一个文档有多个 chunk
- 一个知识库有多个问答会话
- 一个问答会话有多条消息
- 文档列表支持按文件类型区分，例如 `word`、`pdf`、`markdown`
- 支持基于知识库的 RAG 问答与 SSE 流式输出

### 3.3 为什么当前只做到这里

因为完整项目的合理演进顺序是：

1. 先把用户与鉴权打稳
2. 再把知识库主记录打稳
3. 再补文档与 chunk 主链路
4. 最后接入问答、检索和流式输出

这能避免项目一开始就进入过度复杂状态。

## 4. 核心名词表

| 名词 | 说明 |
| --- | --- |
| 用户 | 系统登录主体 |
| 知识库 | 一组文档的容器，例如“前端面试资料库” |
| 文档 | 一份上传的文件或导入的文本 |
| Chunk | 文档切分后的最小检索单元 |
| 会话 | 某个知识库内的一次连续问答上下文 |
| 消息 | 会话中的一条输入或输出 |
| RAG | 先检索、再生成回答 |
| Embedding | 把文本转换成向量表示 |
| SSE | 服务端流式返回文本片段 |

## 5. 技术选型

### 5.1 当前代码已采用

- 后端框架：`NestJS 11`
- 数据库：`MongoDB + Mongoose`
- 鉴权：`JWT accessToken + refreshToken`
- 文档：`Swagger`
- 包管理器：`pnpm`

### 5.2 完整项目预计采用

- 文档存储：对象存储，例如 `Cloudflare R2`
- 向量检索：`MongoDB Atlas Vector Search`
- 模型接入：OpenAI 兼容聊天模型与 Embedding 模型
- AI 编排：`LangChain`

需要明确：

- 上面这些 AI 相关能力属于完整蓝图，不代表当前仓库已具备
- 当前部署文档只要求你准备当前代码真正需要的变量和服务

### 6.2 第二阶段：补上文档主链路

- 新建 `documents`
- 实现用户级文档列表

当前状态：规划中

### 6.3 第三阶段：补上 chunk 与检索基础

- 新建 `document_chunks`

当前状态：规划中

- 新建 `chat_sessions`
- 新建 `chat_messages`
- 接入 RAG JSON 问答

当前状态：规划中

## 8. 当前阶段建议

如果你正在基于当前仓库继续往后做，优先顺序建议是：

1. `documents`
2. `document_chunks`
3. `chat_sessions` / `chat_messages`
4. RAG 检索
5. SSE

这个顺序的好处是：

- 前端文档列表能最早得到支持
- 数据模型的依赖关系自然展开
- 后续 AI 能力能建立在稳定的文档主链路上
